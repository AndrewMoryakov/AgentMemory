# Visual Identity

This document defines the visual design concept for AgentMemory and the broader family of future technical projects around it.

The goal is not only to generate one good image.

The goal is to establish a reusable design language that:

- fits AgentMemory today
- can carry into future projects
- remains recognizable across repositories
- stays grounded in real engineering meaning

## Core Idea

The visual system should present software infrastructure as a **signal-controlled network**.

The real-world inspiration is:

- railway dispatch maps
- signal boards
- route-switching diagrams
- infrastructure control systems

This should **not** become literal train imagery.

The project should borrow the logic of those systems:

- a central control hub
- connected routes
- distributed nodes
- switching points
- visible signals
- a feeling of live coordination

The visual identity can be summarized as:

**signal maps for software**

## Why This Fits AgentMemory

AgentMemory is a shared local runtime that coordinates:

- multiple clients
- multiple surfaces
- multiple providers
- one central memory runtime

That is structurally similar to a dispatch or control network:

- clients enter from different directions
- one core runtime routes and normalizes behavior
- providers sit behind the runtime boundary
- signals, transport, and system state matter

So the design language should not look like generic AI branding.

It should look like infrastructure that is:

- alive
- organized
- routed
- monitored
- controllable

## Visual Metaphor

The main recurring metaphor is:

**a control junction**

Each project in this visual family should feel like:

- one central technical core
- surrounded by modular external nodes
- joined by route lines or signal paths
- presented as an elegant system map

For AgentMemory specifically:

- center: `AgentMemory`
- left: tools and client surfaces
- right: memory providers and backends

This mirrors the actual architecture rather than inventing unrelated symbolism.

## Permanent Visual Traits

These are the traits that should repeat across future projects.

### 1. A strong central core block

Every image should have one visually dominant central block or hub.

This block represents:

- the runtime
- the control layer
- the main system identity

For AgentMemory, that block is the runtime itself.

### 2. Surrounding node tiles

External systems should appear as distinct surrounding nodes.

These nodes represent:

- clients
- tools
- adapters
- providers
- outputs
- companion systems

They should feel modular and separate, not melted into one abstract mass.

### 3. Connection lines as routes

Connections should feel like:

- routes
- signal lines
- switching paths
- technical map lines

This creates the sense that the image is a system topology, not a poster with random icons.

### 4. Signal markers and junctions

The design should include small indicators such as:

- route intersections
- signal points
- direction markers
- status lights
- node indicators

These make the system feel operational and infrastructure-like.

### 5. Subtle pixel structure

Pixel language is a core differentiator.

It should be used intentionally:

- visible square-pixel forms
- crisp blocky geometry
- pixel-grid logic
- retro-digital rhythm

The project should not look like smooth illustration with pixel noise pasted on top.

It should look like real pixel-aware design.

## Style Direction

The intended style is:

- technical
- diagrammatic
- retro-digital
- slightly artistic
- open-source infrastructure
- calm and precise

It should feel like:

**engineering diagrams transformed into poster-art**

It should not feel like:

- startup SaaS branding
- generic AI illustration
- cyberpunk cliché
- cartoon pixel game art

## Pixel Art Policy

If the output is generated as pixel art, these rules should hold.

### What to preserve

- visible square pixels
- crisp edges
- blocky forms
- grid alignment
- flat pixel shading
- modular icon tiles

### What to avoid

- fake pixel filters
- blurry pixel edges
- smooth gradients
- cinematic painterly lighting
- glossy 3D surfaces
- anti-aliased vector look

### Important practical rule

If the image is meant to be pixel art, pixel art must be the **primary style**, not a decorative accent.

## Color System

Use a restrained retro-technical palette.

Recommended base palette:

- `#101418` deep graphite
- `#1A2229` slate black
- `#24303A` steel blue-gray
- `#D9E1E8` cool off-white
- `#8FB9C9` muted cyan
- `#6FD3C1` mint signal accent
- `#E0A85E` amber indicator
- `#6E7C87` muted secondary label

### Palette roles

- background: graphite / deep slate
- structure lines: steel blue / muted cyan
- signal highlights: mint and amber
- text: off-white
- secondary labels: dusty gray-blue

### Palette guidance

- use limited contrast, not rainbow color usage
- accents should feel like system indicators, not decorative candy
- amber should suggest signal lights
- mint/cyan should suggest active routes or system wiring

## Composition Rules

These rules are the core of the reusable visual language.

### Standard composition

- one dominant center block
- left side: inputs, clients, tools, or request surfaces
- right side: providers, outputs, backends, or destinations
- connection lines routing through the center
- labels kept short and technical

### Density

- avoid overcrowding
- prefer 5-10 meaningful nodes over many tiny labels
- leave breathing room around the central block

### Hierarchy

- project title must be immediately readable
- system structure should be understandable at a glance
- smaller node labels can be secondary

### Readability

GitHub social preview is a small target in many contexts.

So:

- title must remain readable at thumbnail size
- subtitle should remain readable when possible
- small labels should support the composition, not dominate it

## AgentMemory-Specific Interpretation

For AgentMemory, the image should communicate:

- many clients on one side
- many providers on the other side
- one shared runtime in the center
- system routing and runtime coordination

Useful left-side node examples:

- `Codex`
- `Claude`
- `Scripts`
- `MCP Clients`
- `Browser UI`

Useful right-side node examples:

- `mem0`
- `localjson`

Important:

`mem0` should not be presented as the only possible backend in the visual language.

## What Makes This Identity Distinct

The distinctiveness should come from the combination of:

- a real-world control-network metaphor
- central runtime composition
- modular node tiles
- signal-line routing
- disciplined pixel structure
- calm infra palette

This is stronger than relying on:

- mascots
- robots
- generic AI symbols
- arbitrary decoration

## What To Avoid

Avoid these recurring mistakes:

- glowing brain imagery
- humanoid robots
- overly purple AI branding
- generic cyberpunk city scenes
- smooth corporate SaaS illustration
- fantasy scenes unrelated to software structure
- literal trains or literal railway station imagery
- cluttered dashboards pretending to be branding
- childish arcade-game aesthetics

The metaphor is infrastructure logic, not transport nostalgia.

## How To Tune The Style

This design system should be adjustable depending on the need.

### If you want it more serious

- reduce glow
- reduce decorative effects
- reduce number of node labels
- increase diagram clarity
- use darker slate backgrounds
- make the center block more dominant

This is useful for:

- repository social preview cards
- architecture visuals
- release images

### If you want it more artistic

- increase visual rhythm in route lines
- add faint pixel-grid depth
- add subtle signal glow around junctions
- use more deliberate spatial composition
- let the central hub feel more atmospheric

This is useful for:

- launch visuals
- social posts
- presentation covers

### If you want it more pixel-heavy

- make every entity a clearly separate pixel icon tile
- enforce visible square pixels
- remove smooth haze entirely
- use flat shading only
- keep geometry more rigid

This is useful for:

- distinctive GitHub branding
- image generator stability
- consistent retro-digital identity

### If you want it less pixel-heavy

- keep the composition and metaphor
- keep the route-map structure
- keep the color system
- reduce visible pixel block size
- retain only a subtle raster or digital-grid trace

This is useful for:

- documentation illustrations
- polished web headers
- general product design assets

## Prompting Guidance For Image Generators

When prompting image models, separate three concerns:

### 1. Meaning

Explain the actual product:

- one shared runtime
- many clients
- many providers
- local developer infrastructure

### 2. Metaphor

Use:

- control network
- dispatch map
- signal junction
- software routing board

Do not over-emphasize literal transport imagery.

### 3. Style lock

If pixel art is required, explicitly state:

- true pixel art
- visible square pixels
- flat pixel shading
- grid-aligned composition
- no smooth illustration

This is necessary because many models default to smooth digital art.

## Reusable Prompt Structure

Use this structure when generating future project images:

1. What the project does
2. The core visual metaphor
3. Required composition
4. Required text
5. Style constraints
6. Color palette
7. Negative prompt

This is more reliable than one vague “make it cool” prompt.

## How To Adapt This Identity To Other Projects

The visual system should stay consistent, but each project should swap:

- the central block label
- the surrounding node types
- the direction of flow
- the specific domain wording

Examples:

- memory runtime project: clients -> runtime -> providers
- orchestration project: agents -> orchestrator -> tools
- routing project: sources -> router -> targets
- indexing project: inputs -> indexer -> stores/search

The visual DNA stays the same even when the project theme changes.

## Default Recommendation For AgentMemory

If no special requirement exists, use this default:

- central `AgentMemory` control hub
- left-side client nodes
- right-side provider nodes
- signal-map routing lines
- visible but disciplined pixel structure
- calm graphite/cyan/mint/amber palette
- strong title readability
- slight artistic atmosphere, but still clearly technical

That is the current best expression of the project.
